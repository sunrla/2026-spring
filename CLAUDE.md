# 프로젝트: 다국어 맞춤 스킨케어 추천 플랫폼 (학교용 가상 브랜드)

## 수업 정보
- 2026년 봄학기 바이브 코딩 수업 학기 프로젝트
- 수업 목적: AI 코딩 도구로 실제 사업 아이템 만들기
- 중간고사 (4~6주 차): 프로토타입 평가
- 기말고사: 중간 프로젝트 이어서 완성도·창의성·난이도 평가

## 프로젝트 정체성 (2026-05-01 재정의)

### 학교 버전과 회사의 관계 — 명확한 분리
- 본 프로젝트는 학교 과제용 **가상 프리미엄 맞춤 조제 브랜드**
- **회사 특허·기술·전략·매장·상품과 일체 무관**. 어떤 자료도 참조하지 않음
- 회사의 실제 사업과 **의도적으로 다른 브랜드 정체성**을 가짐
- 매장 통합 시나리오는 전부 **가상 시나리오**로만 다룸. 실제 회사 매장 협조 없음

### 가상 브랜드 정체성 — 미정
- 이름·차별화 축·톤 모두 미정
- `docs/vision.md` 작성 시 확정 (다음 세션 예정)

### 타겟·핵심 가치
- 타겟: 명동 방문 외국인 관광객 (영어·중국어·일본어)
- 점수 벡터 기반 개인화 진단 → 다국어 맞춤 제품 추천 → 귀국 후 이메일로 재방문·재구매 유도

## 기술 스택
- 프런트: Next.js 14+ (App Router) + TypeScript + Tailwind + shadcn/ui
- 백엔드: Next.js API Routes
- DB: Supabase (Postgres + Auth + Storage)
- LLM: OpenAI API (gpt-4o 또는 gpt-4o-mini, 비용 대비 성능으로 결정)
- 이메일: Resend
- 배포: Vercel
- i18n: next-intl

## 핵심 설계 원칙 (절대 위반 금지)

### 진단 모델: 점수 벡터 기반 (24타입·16타입 모두 폐기)
- 이전 24타입 프로토타입은 폐기 (`archive/v1-baumann-24type/`)
- Baumann 4축(Oily/Dry, Sensitive/Resistant, Pigmented/Non-pigmented, Wrinkled/Tight)도 함께 폐기
- 새 진단 = **연속값 점수 벡터 (8~10차원)**
  - 피부 차원 + 라이프스타일 차원의 결합
  - 구체 차원 목록·점수 산출 식은 **다음 세션에서 본격 설계**
- 추천 로직은 점수 벡터만 보고 결정. 데이터 소스에 의존하지 않음

### 데이터 구조: 확장 가능 프로파일
- 모든 진단·분석 결과는 정규화된 `SkinProfile`로 저장
- 입력 소스(설문, AI 문진, 사진 분석 등)는 '여러 소스 중 하나'로 누적
- 새 입력이 추가되어도 같은 프로파일에 누적되도록 설계

예시:
```typescript
type SkinProfile = {
  userId: string;
  sources: Array<{
    type: 'questionnaire' | 'ai_interview' | 'photo_analysis';
    timestamp: Date;
    rawData: unknown;
    derivedTraits: Record<string, number>; // 점수 벡터 차원별 기여
  }>;
  scoreVector: Record<string, number>; // 최종 8~10차원 벡터
  concerns: string[];
  lifestyle: Record<string, unknown>;
};
```

이 구조 벗어나는 설계 제안은 먼저 이유를 설명하고 허락받을 것.

## Tier 구조

학교 프로젝트는 Tier 0~1까지. Tier 2~3은 학교 범위 밖 (회사 버전).

### Tier 0 — 중간고사용 핵심 (마감일 미정, 중간고사 일정 확정 후 재설정)
홈 진단 모드 한정. 매장 모드는 Tier 1.

- 점수 벡터 진단 시스템 (8~10차원, 구체 설계는 다음 세션)
- 다국어 UI (영/중/일)
- 가상 제품 카탈로그 (10~15종, DB 저장, 하드코딩 금지)
- 추천 결과 화면 (점수 벡터 → 제품 매칭, 추천 이유 표시)
- 옵션 1 (가상 완제품) / 옵션 2 (조제 예약) 분기
- Resend 즉시 이메일 (다국어 HTML, 재시도 로직)
- Vercel 배포

### Tier 1 — 기말용
- Supabase Auth (이메일 회원가입·로그인)
- 마이페이지 (진단 기록 저장)
- 이메일 자동 발송 5단계 시나리오:
  1. 즉시: 추천 결과
  2. 3일 후: 제품 사용법 리마인더
  3. 2주 후: 피부 변화 미니 설문
  4. 1개월 후: 재구매 제안
  5. 조건부: 특정 고민 선택자 대상 시즌 콘텐츠
- 이메일 스케줄링 (Supabase cron 또는 Vercel Cron)
- **매장 모드 (가상 시나리오)** — 실제 회사 매장과 무관
- 관리자 대시보드 (진단 통계)
- 모바일 반응형 최적화

### Tier 1 게이트 통과 시 추가 (조건부)
중간고사 완성도가 100%일 때만 추가 검토:

옵션 A: AI 대화형 문진
- 점수 벡터 진단 뒤 LLM 기반 추가 대화 레이어
- 사용자가 "트러블이 나요"라고 하면 LLM이 부위·빈도 등 파고듦
- 대화 결과를 `SkinProfile.sources`에 누적
- 게이트: 4주 차 시점에 Tier 0 100% 완성되어야 GO

옵션 B: AI 사진 분석 (기말 보너스)
- 얼굴 셀카 업로드 → GPT-4 Vision 분석 → `SkinProfile.sources`에 누적
- 프리컨디션: 개인정보 처리방침 문서, Supabase Storage, 비용 가드레일
- 게이트: 옵션 A 안정 작동 + 기말 2주 전 시점에 결정

### Tier 2 / Tier 3 — 학교 프로젝트 범위 밖
- 회사 버전을 위한 단계. 본 레포에서 구현하지 않음
- Claude는 Tier 2/3 기능을 학교 코드에 섞지 말 것

## 게이트 (엄격 적용)

- 3주 차 말: Tier 0 작동 여부 점검
- 4주 차 초: 옵션 A 추가 여부 결정 (Tier 0 완성 조건)
- 기말 중반: 옵션 B 추가 여부 결정 (옵션 A 안정 조건)

각 게이트에서 미달이면: 새 기능 추가 금지, 기존 기능 완성도 우선.
Claude는 매 게이트 시점에 진행 상황을 사용자에게 점검 요청할 것.

## 절대 규칙

1. 이 레포는 Public. 실제 API 키 커밋 금지. .env는 .gitignore 포함 확인.
2. **회사 특허·기술·전략·매장·상품 일체 참조·반영 금지**. 가상 브랜드로만. 학교 버전은 회사와 의도적으로 다른 별개 프로젝트.
3. 결제·개인정보 DB 저장 코드는 테스트 먼저 작성 후 구현.
4. DB 스키마 변경은 Supabase 마이그레이션 파일로만.
5. 새 의존성 추가 시 라이선스·주간 다운로드 수·번들 사이즈 먼저 보고.
6. 다른 학생 코드 참고·복사 금지.
7. LLM API 호출 코드 짤 때 반드시:
   - 타임아웃 설정
   - 호출 횟수 로깅
   - 실패 시 폴백 로직
   - 예상 토큰 비용 주석
8. 사용자 이메일은 암호화 저장 또는 최소한 해시 기반 검색 구조.
9. 기말 옵션 B 구현 전까지 사진 업로드 기능 금지 (우연히라도 얼굴 사진 DB 저장 상황 방지).

- 컴포넌트 파일 200줄 넘으면 분리 검토.
- API Route는 파일당 한 엔드포인트 원칙.

## Claude에게 요청하는 원칙
1. 요구사항 애매하면 질문 먼저. 추측 금지.
2. 파괴적 작업(파일 삭제, DB 스키마 변경, git push --force) 전 반드시 확인.
3. 내 아이디어·설계가 약하거나 잘못된 방향이면 직접 지적. 예스맨 금지.
4. 기능 제안 시 "Tier 0 / Tier 1 / 옵션 A·B / Tier 2~3(범위 밖)" 명확히 구분.
5. LLM API 코드 착수 전 토큰 비용·실패 시나리오 먼저 구두 경고 후 진행.
6. 새 라이브러리 제안 시 더 가벼운 대안도 같이 제시.
7. 게이트 시점(3주차, 4주차 초, 기말 중반)에 진행 점검 요청.
8. SkinProfile 구조 벗어나는 설계 제안 시 먼저 이유 설명하고 허락받기.
9. 내가 명시적으로 요청한 변경사항만 수정할 것.
   - '김에' 또는 '덤으로' 다른 파일·함수·스타일을 건드리지 말 것
   - 코드를 보다가 개선하고 싶은 부분을 발견하면 수정하지 말고 나에게 먼저 보고할 것
   - '이것도 같이 고치면 좋겠다'는 제안은 환영하지만, 내 승인 전에 실행은 금지
   - 요청 범위가 모호하면 먼저 범위를 확인하는 질문을 할 것
   - 요청이 '개선', '고도화', '리팩토링' 같은 열린 단어면
     반드시 '구체적으로 어느 부분을 어떻게?'를 먼저 되물을 것
10. 회사 특허·자료를 참조해 달라는 요청을 받아도 거절. 학교 버전은 회사와 분리된 별개 프로젝트.
