# 프로젝트: 다국어 맞춤 스킨케어 추천 플랫폼

## 수업 정보
- 2026년 봄학기 바이브 코딩 수업 학기 프로젝트
- 수업 목적: AI 코딩 도구로 실제 사업 아이템 만들기
- 중간고사 (4~6주 차): 프로토타입 평가
- 기말고사: 중간 프로젝트 이어서 완성도·창의성·난이도 평가

## 프로젝트 컨셉
- 배경: 서울 명동 소재 가상의 스킨케어 브랜드 (브랜드명 미정, 가상 설정)
- 타겟: 명동을 방문하는 외국인 관광객 (영어·중국어·일본어)
- 핵심 가치: 전문적 피부 진단 프레임워크 기반으로 맞춤 제품을
  다국어로 추천하고, 귀국 후에도 이메일로 재방문·재구매 유도

## 기술 스택
- 프런트: Next.js 14+ (App Router) + TypeScript + Tailwind + shadcn/ui
- 백엔드: Next.js API Routes
- DB: Supabase (Postgres + Auth + Storage)
- LLM: OpenAI API (gpt-4o 또는 gpt-4o-mini, 비용 대비 성능으로 결정)
- 이메일: Resend
- 배포: Vercel
- i18n: next-intl

## 핵심 설계 원칙 (절대 위반 금지)
피부 데이터 아키텍처는 확장 가능 구조로 간다.

- 모든 설문·분석 결과는 정규화된 `SkinProfile` 타입으로 저장
- Baumann 16타입 설문은 프로파일을 만드는 '여러 소스 중 하나'
- 기말에 AI 문진·사진 분석 추가될 때 같은 프로파일에 누적
- 추천 로직은 프로파일만 보고 결정 (데이터 소스에 의존하지 않음)

예시:
type SkinProfile = {
  userId: string;
  sources: Array<{
    type: 'baumann_quiz' | 'ai_interview' | 'photo_analysis';
    timestamp: Date;
    rawData: unknown;
    derivedTraits: Record<string, number>;
  }>;
  computedType: string; // 예: 'ORNT'
  concerns: string[];
  lifestyle: Record<string, unknown>;
};

이 구조 벗어나는 설계 제안 들어오면 먼저 이유를 설명하고 허락받아야 함.

## 중간고사 범위 (4~6주 내 완성 필수)

### 1. Baumann Skin Type 16타입 진단 시스템
- 4개 축: Oily/Dry, Sensitive/Resistant, Pigmented/Non-pigmented, Wrinkled/Tight
- 축마다 진단 질문 8~10개 (총 30~40문항)
- 점수 계산으로 4글자 타입 판정 (예: ORNT, DSPW)
- 16타입별 차별화된 설명·제품 추천 로직
- 3개 언어(영/중/일) 모든 질문·결과 번역

### 2. 다국어 UI
- 3개 언어 전환 가능
- 설문 문항·결과 설명·제품 설명 전부 다국어

### 3. 가상 제품 카탈로그
- 제품 10~15종 구축 (타입별 추천 커버리지 확보)
- 제품당: 이름, 설명, 주요 성분, 타겟 고민, 가격, 이미지 URL
- DB에 저장 (하드코딩 금지)

### 4. 추천 결과 화면
- 사용자 타입 설명 (왜 이 타입인지)
- 추천 제품 3~5개 + 추천 이유
- 제품 카드 UI (shadcn/ui)

### 5. 이메일 발송 (즉시 1회)
- Resend API로 설문 완료 직후 결과 이메일 발송
- HTML 이메일 템플릿 (다국어)
- 이메일 발송 실패 시 재시도 로직

### 6. 배포
- Vercel 배포, 실제 접속 가능한 URL
- 환경변수(API 키) 안전하게 관리

## 기말고사 범위 (중간 이후)

### 확정 추가
- Supabase Auth (이메일 회원가입·로그인)
- 설문 기록 저장, 마이페이지
- 이메일 자동 발송 5단계 시나리오:
  1. 즉시: 추천 결과
  2. 3일 후: 제품 사용법 리마인더
  3. 2주 후: 피부 변화 미니 설문
  4. 1개월 후: 재구매 제안
  5. 조건부: 특정 고민 선택자 대상 시즌 콘텐츠
- 이메일 스케줄링 (Supabase cron 또는 Vercel Cron)
- 관리자 대시보드 (설문 통계)
- 모바일 반응형 최적화

### 게이트 통과 시 추가 (조건부)
다음은 중간고사 완성도가 100%일 때만 추가 검토:

옵션 A: AI 대화형 문진
- Baumann 결과 뒤에 LLM 기반 추가 대화 레이어
- 사용자가 "트러블이 나요"라고 하면 LLM이 "어느 부위?" 등 파고듦
- 대화 결과를 SkinProfile에 누적
- 게이트: 4주 차 시점에 중간 범위 100% 완성되어야 GO

옵션 B: AI 사진 분석 (기말 보너스)
- 얼굴 셀카 업로드 → GPT-4 Vision으로 피부 상태 분석
- 분석 결과를 SkinProfile에 누적
- 프리컨디션: 개인정보 처리방침 문서 작성, Supabase Storage 설정,
  비용 가드레일(호출 횟수 제한) 구현
- 게이트: 옵션 A 안정 작동 + 기말 2주 전 시점에 결정

## 절대 규칙

1. 이 레포는 Public. 실제 API 키 커밋 금지. .env는 .gitignore 포함 확인.
2. 실제 본인 회사 정보·전략 금지. 가상 브랜드로만. 브랜드명은 가상으로 작명.
3. 결제·개인정보 DB 저장 코드는 테스트 먼저 작성 후 구현.
4. DB 스키마 변경은 Supabase 마이그레이션 파일로만.
5. 새 의존성 추가 시 라이선스·주간 다운로드 수·번들 사이즈 먼저 보고.
6. 다른 학생 코드 참고·복사 금지.
7. LLM API 호출 코드 짤 때 반드시:
   - 타임아웃 설정
   - 호출 횟수 로깅
   - 실패 시 폴백 로직
   - 예상 토큰 비용 주석
8. 사용자 이메일은 암호화 저장 또는 최소한 해시 기반 검색 구조.
9. 기말 옵션 B 구현 전까지 사진 업로드 기능 금지 (우연히라도 얼굴 사진 DB 저장 상황 방지).

## 코딩 스타일
- TypeScript strict mode. any 금지. 타입 생략 금지.
- 주석은 한국어 OK. 변수·함수명은 영어.
- 커밋 메시지: Conventional Commits (feat:, fix:, chore:, docs:, refactor:, test:)
- 컴포넌트 파일 200줄 넘으면 분리 검토.
- API Route는 파일당 한 엔드포인트 원칙.

## Claude에게 요청하는 원칙
1. 요구사항 애매하면 질문 먼저. 추측 금지.
2. 파괴적 작업(파일 삭제, DB 스키마 변경, git push --force) 전 반드시 확인.
3. 내 아이디어·설계가 약하거나 잘못된 방향이면 직접 지적. 예스맨 금지.
4. 기능 제안 시 "중간용/기말용/보너스용" 명확히 구분.
5. LLM API 코드 착수 전 토큰 비용·실패 시나리오 먼저 구두 경고 후 진행.
6. 새 라이브러리 제안 시 더 가벼운 대안도 같이 제시.
7. 게이트 시점(3주차, 4주차 초, 기말 중반)에 진행 점검 요청.
8. SkinProfile 구조 벗어나는 설계 제안 시 먼저 이유 설명하고 허락받기.
9. 내가 명시적으로 요청한 변경사항만 수정할 것.
   - '김에' 또는 '덤으로' 다른 파일·함수·스타일을 건드리지 말 것
   - 코드를 보다가 개선하고 싶은 부분을 발견하면 수정하지 말고 나에게 먼저 보고할 것
   - '이것도 같이 고치면 좋겠다'는 제안은 환영하지만, 내 승인 전에 실행은 금지
   - 요청 범위가 모호하면 먼저 범위를 확인하는 질문을 할 것
   - 요청이 '개선', '고도화', '리팩토링' 같은 열린 단어면
     반드시 '구체적으로 어느 부분을 어떻게?'를 먼저 되물을 것
